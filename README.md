# Lifecycle Ticket

Сбор статистики по тикетам helpdesk из трёх задач в едином скрипте.

## Требования

```bash
pip install -r requirements.txt
```

Заполни `.env` по образцу `.env.example`. Положи рядом `firm-amp-464221-g0-10e1752037bb.json` (сервис-аккаунт Google).

---

## Запуск

```bash
python run.py --task <задача> --from <дата> --to <дата> [опции]
```

### Задачи (`--task`)

| Задача | Что собирает | Google Sheets |
|---|---|---|
| `all` | Lifecycle каждого тикета + кол-во передач + кол-во постов | gid=2050386850 |
| `lifecycle` | Только lifecycle тикетов | gid=640640423 |
| `cloud-stats` | Месячная статистика: кол-во тикетов и передач на Вторую линию | gid=1877381753 |
| `cloud-moves` | Список тикетов с передачами на Вторую линию + autorouting | gid=1373632618 |

### Флаги

| Флаг | Описание | Применимо |
|---|---|---|
| `--from YYYY-MM-DD` | Дата начала периода (включительно) | все задачи |
| `--to YYYY-MM-DD` | Дата конца периода | все задачи |
| `--tag <slug>` | Фильтр по AI-тегу на первом посте тикета (default: `cloud_first`) | `all`, `lifecycle` |
| `--ticket <id>` | Отладка: собрать только один конкретный тикет | `all`, `lifecycle` |
| `--no-sheet` | Не писать в Google Sheets, только вывод в консоль | все задачи |

> Для `cloud-stats` и `cloud-moves` граница `--to` исключающая: `--to 2026-05-01` означает «до 1 мая, не включая».

---

## Примеры

```bash
# Полный сбор за апрель 2026 — lifecycle + передачи + посты
python run.py --task all --from 2026-04-01 --to 2026-04-30

# Только lifecycle, без записи в таблицу (для отладки)
python run.py --task lifecycle --from 2026-04-01 --to 2026-04-30 --no-sheet

# Один тикет — проверить как он разобрался
python run.py --task all --from 2026-04-01 --to 2026-04-30 --ticket 2835619 --no-sheet

# Месячная статистика по cloud-тикетам за год
python run.py --task cloud-stats --from 2025-04-01 --to 2026-05-01

# Список тикетов с передачами на Вторую линию
python run.py --task cloud-moves --from 2025-04-01 --to 2026-05-01
```

## Колонки (`--task all`)

| Колонка | Описание |
|---|---|
| `ticket_id` | URL тикета: `https://hp.beget.ru/ticket/<id>` |
| `start_at` | Время создания тикета (UTC+3) |
| `end_at` | Время закрытия — последний `feedback` без ответа клиента в течение 24ч (UTC+3) |
| `время_жизни_сек` | Полное время жизни тикета: `end_at − start_at` |
| `ожидание_ответа_саппорта_сек` | Суммарное время, пока клиент ждал ответа саппорта (тикет в работе у сотрудника) |
| `ожидание_ответа_клиента_сек` | Суммарное время, пока саппорт ждал ответа клиента (статус `feedback` ≤ 24ч) |
| `время_жизни_мин` | `время_жизни_сек / 60` — формула в Sheets |
| `ожидание_ответа_саппорта_мин` | `ожидание_ответа_саппорта_сек / 60` — формула в Sheets |
| `ожидание_ответа_клиента_мин` | `ожидание_ответа_клиента_сек / 60` — формула в Sheets |
| `число_итераций` | Кол-во циклов «клиент написал → саппорт ответил → feedback». Минимум 1 |
| `число_постов_пользователя` | Кол-во постов от клиента (`employee_id IS NULL`) |
| `число_постов_саппорта` | Кол-во постов от сотрудников (`employee_id IS NOT NULL`) |
| `кол_во_передач` | Передачи на `Cloud:Вторая линия`: 0 = не передавался, 1 = ручная, >1 = был autorouting |
| `кол_во_передач_вручную` | Передачи на Вторую линию сделанные вручную (не autorouting) |
| `lifecycle_history` | Фазы тикета: `ожидание Xс → в работе Xс (Имя) → ... → тикет закрыт` |

### Как определяется autorouting

Передача считается autorouting если в `journal` запись имеет:
- `author = 'VM support'` и `employee_id = 716`
- или `author = 'Сотрудник Для-Скрипта-Клюкова А.Е.'` и `employee_id = 842`
